# App Router Security Patterns Reference

Load this when auditing Next.js App Router code (`app/` directory in Next.js 13+).

## The model in one paragraph

App Router has three execution boundaries: **Server Components** (default, run server-side only, can read secrets and databases), **Client Components** (marked `"use client"`, ship to the browser, can't read server secrets), and **Server Actions** (`"use server"`, RPC endpoints generated by Next, callable from the client). Layouts, pages, route handlers, and middleware are all Server Components / server-side by default. The boundary between Server and Client is where most security bugs live.

## Server / Client boundary

### Prop serialization

Every prop passed from a Server Component to a Client Component is JSON-serialized. This means:

- Secrets in props leak to the browser.
- Functions can't be passed (Server Actions are the workaround).
- Circular references break.

```tsx
// app/dashboard/page.tsx — Server Component
import { db } from '@/lib/db';
import { DashboardClient } from './DashboardClient';

export default async function DashboardPage() {
  const user = await db.users.findCurrent();
  // BAD — passes entire user record including hashed password, internal IDs
  return <DashboardClient user={user} />;
}

// GOOD — pass only what client needs
return <DashboardClient
  displayName={user.displayName}
  avatarUrl={user.avatarUrl}
/>;
```

The fix is to project to a "public DTO" before crossing the boundary. Patterns:

```ts
// lib/dtos.ts
export function publicUserDto(user: User) {
  return {
    id: user.id,
    displayName: user.displayName,
    avatarUrl: user.avatarUrl,
    // explicitly NOT: passwordHash, emailHash, internalNotes, billingAddress
  };
}
```

### `server-only` and `client-only` packages

```ts
// lib/auth-server.ts
import 'server-only';  // throws at build if imported from a Client Component
import { headers } from 'next/headers';

export async function getServerSession() { /* ... */ }
```

```ts
// lib/browser-storage.ts
import 'client-only';
export function persistDraft(...) { /* uses localStorage */ }
```

These guard rails turn accidental imports into build failures. Use them for any module touching secrets, env vars, DB clients, or browser-only APIs.

### Detection — secrets crossing the boundary

```bash
# Files that touch secrets
grep -rln 'process\.env\.\(DATABASE_URL\|SECRET_KEY\|STRIPE_SECRET\|JWT_SECRET\|.*_API_KEY\|.*_PRIVATE_KEY\)' app/ src/ lib/

# Among those, which are imported by Client Components?
# Trace each file: does any "use client" file import from it?
grep -rln "from .*/lib/db" app/ | xargs grep -l '"use client"' 2>/dev/null
```

Any "use client" file in the import graph of a secret-handling module is a finding — even if the specific export doesn't reference the secret directly. The bundler doesn't always tree-shake reliably enough to trust.

## Server Actions

### Anatomy

```ts
'use server';

import { auth } from '@/lib/auth';
import { revalidatePath } from 'next/cache';

export async function deletePost(formData: FormData) {
  const session = await auth();
  if (!session?.user) throw new Error('Unauthorized');
  
  const postId = formData.get('postId') as string;
  // ... validate postId, check ownership, delete
  
  revalidatePath('/posts');
}
```

The function is callable from any Client Component via:

```tsx
'use client';
import { deletePost } from './actions';

<form action={deletePost}>
  <input type="hidden" name="postId" value={post.id} />
  <button type="submit">Delete</button>
</form>

// or
<button onClick={() => deletePost(formData)}>
```

### Auth check pattern

Every Server Action that does anything sensitive starts with auth + authz:

```ts
'use server';

import { auth } from '@/lib/auth';
import { z } from 'zod';

const DeletePostSchema = z.object({
  postId: z.string().uuid(),
});

export async function deletePost(formData: FormData) {
  // 1. AuthN
  const session = await auth();
  if (!session?.user) throw new Error('Unauthorized');
  
  // 2. Input validation
  const { postId } = DeletePostSchema.parse({
    postId: formData.get('postId'),
  });
  
  // 3. AuthZ — does this user own the post?
  const post = await db.posts.findById(postId);
  if (!post) throw new Error('Not found');
  if (post.authorId !== session.user.id) throw new Error('Forbidden');
  
  // 4. Perform action
  await db.posts.delete(postId);
  
  // 5. Revalidate
  revalidatePath('/posts');
}
```

### Common Server Action vulnerabilities

**Missing auth check:**
```ts
'use server';
export async function adminAction(userId: string) {
  // ⚠ no auth check
  await db.users.makeAdmin(userId);
}
```
Anyone can call this from their browser console:
```js
fetch('/dashboard', { method: 'POST', headers: { 'Next-Action': '<action-id>' }, body: '["target-user-id"]' })
```

**User ID from input, not session:**
```ts
'use server';
export async function updateProfile(userId: string, data: ProfileData) {
  // ⚠ trusts caller's claim about whose profile to update
  await db.users.update(userId, data);
}
```
Always derive `userId` from `auth()`, never from args.

**No input validation:**
```ts
'use server';
export async function createPost(data: any) {
  // ⚠ stores attacker-controlled fields including `published`, `featured`, `authorId`
  await db.posts.create(data);
}
```
Use Zod (or yup, valibot) to validate and strip.

**Revealing details in errors:**
```ts
'use server';
export async function lookupUser(email: string) {
  const user = await db.users.findByEmail(email);
  if (!user) throw new Error(`User not found: ${email}`);
  // ... error propagates to client; reveals user enumeration
}
```
Generic errors to the client; detail in server logs.

### Server Action IDs

Server Actions get an automatically-generated ID. The ID is in the bundle but isn't a secret — anyone can discover and call any Server Action. **Treat every Server Action as a public RPC endpoint and apply auth/authz like you would to a REST endpoint.**

### CSRF on Server Actions

Next.js applies Same-Origin restrictions to Server Action calls via the framework's encoding. Programmatic POSTs with the Server Action protocol headers will be checked. But:
- If you call Server Actions from non-browser contexts (mobile, server-side), the protection model changes.
- The protection is "origin must match" — if you have other-origin embeds (subdomains, etc.) that need to call Server Actions, you'll need to handle CORS-like restrictions explicitly.

## Middleware

### What middleware can and can't do

`middleware.ts` runs on the Edge runtime (or Node, depending on config) before any Server Component or Route Handler. It can:
- Rewrite URLs
- Redirect
- Set request/response headers
- Return early with a response

It can't:
- Read request bodies (in most configurations)
- Access Node APIs unless on the Node runtime explicitly

### Auth check pattern

```ts
import { NextResponse, NextRequest } from 'next/server';

export async function middleware(request: NextRequest) {
  // Check protected paths
  const isProtected = ['/dashboard', '/admin'].some(p => 
    request.nextUrl.pathname.startsWith(p)
  );
  if (!isProtected) return NextResponse.next();
  
  // Check auth (e.g., JWT in cookie)
  const session = request.cookies.get('__Host-session')?.value;
  if (!session || !(await verifySession(session))) {
    const loginUrl = new URL('/login', request.url);
    loginUrl.searchParams.set('callbackUrl', request.nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }
  
  return NextResponse.next();
}

export const config = {
  matcher: [
    '/dashboard/:path*',
    '/admin/:path*',
  ],
};
```

### Middleware-related CVEs

**CVE-2025-29927 (March 2025):** A header (`x-middleware-subrequest`) could be set by the client to bypass middleware entirely. Fixed in Next 14.2.25 / 15.2.3. Audit:
1. Next.js version is at least the patched version.
2. Middleware is not the ONLY enforcement — every route handler / Server Action still has its own auth check.

The lesson: middleware is convenience, not a security boundary. Always defense-in-depth.

### Matcher gotchas

The `matcher` array is exact in unexpected ways:
- `/admin` doesn't match `/admin/users` unless you use `/admin/:path*`.
- `/admin/:path*` matches `/admin/x` but NOT `/admin` (the empty path). Use `/admin{/:path*}?` or list both.
- Static files in `/public/` are exempt by default; some sensitive assets land there by accident.

Tests:
```ts
// Add a smoke test in CI that hits every protected path without auth and expects redirect
const protectedPaths = ['/dashboard', '/dashboard/x', '/admin', '/admin/users'];
for (const p of protectedPaths) {
  const r = await fetch(`http://localhost:3000${p}`, { redirect: 'manual' });
  expect(r.status).toBe(307);  // or 308 / 302 depending on config
}
```

## Route Handlers

`app/.../route.ts` files export `GET`, `POST`, etc. as Web Standard handlers:

```ts
// app/api/users/[id]/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { auth } from '@/lib/auth';

export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  
  // Auth + authz
  if (session.user.id !== params.id && !session.user.isAdmin) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }
  
  const user = await db.users.findById(params.id);
  return NextResponse.json(user);
}
```

Treat Route Handlers exactly like REST endpoints — apply all checks from `saas-security-pack/saas-api-security`.

## Cache poisoning awareness

Next caches aggressively (Server Component output, fetch, etc.). Bugs:

- **Per-user data cached at request level** — if cache key doesn't include user context, one user's cache hit returns to another.
  ```ts
  // BAD
  const userData = await fetch('/api/me');  // cached by URL only
  
  // GOOD
  const userData = await fetch('/api/me', { cache: 'no-store' });
  // or
  const userData = await fetch('/api/me', { next: { tags: [`user-${userId}`] } });
  ```
- **Stale auth state in middleware** — cached middleware decisions can leak across sessions.

For per-user pages, use dynamic rendering (no caching) or per-user cache tags. Audit `revalidate` settings on routes that touch user-specific data.

## Audit checklist for App Router

1. Every Server Action: auth + authz + input validation.
2. Every Route Handler: same.
3. No Server-only modules imported by Client Components.
4. Secrets only referenced from server-side code paths.
5. Middleware not the sole auth check.
6. `next/image` remote patterns specific.
7. `NEXT_PUBLIC_` prefix only on truly public values.
8. Build output grep clean for accidental secret leaks.
9. Next.js version is current with security patches.
10. Per-user pages have appropriate cache settings.
